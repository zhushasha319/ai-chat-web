
## 千问 (Qwen) 本地后端集成说明

本项目包含一个简单的 Express 服务器 (`server.js`) 用于把前端 SSE 请求代理到阿里云千问 DashScope API，并转换为接近 OpenAI 的增量格式。

### 1. 安装依赖

```bash
npm install
```

### 2. 配置环境变量

在项目根目录创建 `.env` 文件（不会提交到 git，需自行添加 .gitignore 条目若未存在）：

```bash
QWEN_API_KEY=你的千问APIKey
```

如果你不想在后端写死，也可以在前端设置页填写 API Key（会覆盖后端环境变量，仅保存在浏览器 localStorage）。

### 3. 启动后端

```bash
node server.js
```

启动后默认监听 `http://localhost:3001`，主要路由：

- `POST /api/chat/stream` 进行流式对话（SSE）
- `GET /api/debug/stream` 本地调试流是否正常
- `GET /api/health` 健康检查

### 4. 启动前端

```bash
npm run dev
```

开发模式下前端会向 `/api/chat/stream` 发起请求。若你没有设置 Vite 代理，可在设置页把 `API Base` 改成 `http://localhost:3001/api`。

### 5. 设置页字段说明

| 字段 | 作用 |
|------|------|
| API Base | 后端基础路径，默认 `/api` 方便通过 devServer 代理；也可填完整地址 `http://localhost:3001/api` |
| 模型 | 千问模型名，如 `qwen-turbo` / `qwen-plus` 等 |
| API Key | 可选，填写后以 `Authorization: Bearer xxx` 方式直传后端覆盖环境变量 |
| 系统提示词 | 若当前对话还没有 system 消息，会自动注入为第一条 |

### 6. 前端到后端数据流说明

1. 组件 `ChatRoom.vue` 调用 `useSSE`，向 `${apiBase}/chat/stream` 发送 POST 请求。
2. 后端读取 `messages`、`model`，把它们转发到 DashScope：`https://dashscope.aliyuncs.com/api/v1/services/aigc/text-generation/generation`。
3. DashScope 的 SSE 响应被解析为多次 `data: { choices: [{ delta: { content } }] }` 发送到前端。
4. 前端逐块更新当前 assistant 消息。

### 7. 常见问题排查

| 问题 | 可能原因 | 解决 |
|------|----------|------|
| 400 缺少千问 API Key | 环境变量或前端都没设置 | 在 `.env` 写 `QWEN_API_KEY` 或在设置页填 API Key |
| 没有流内容 / 一次性返回 | 服务器未设置正确 SSE 头 | 确认 `Content-Type: text/event-stream` 且关闭代理缓冲 |
| CORS 报错 | 直接跨域访问 3001 且后端未启动/端口错误 | 确认后端已启动，或配置 Vite 代理 |
| JSON 解析失败日志 | 上游某些行不是 JSON | 可忽略（普通注释/心跳） |

### 8. 可选：Vite 代理示例

可以在 `vite.config.ts` 里添加：

```ts
server: {
	proxy: {
		'/api': {
			target: 'http://localhost:3001',
			changeOrigin: true,
		}
	}
}
```

这样 `API Base` 用默认 `/api` 即可。

### 9. 性能与流式增量更新说明

本项目针对 AI 回复的流式渲染做了几处优化，确保在模型输出 token 较快或消息较长时依然流畅：

1. requestAnimationFrame 合并（前端帧级合并）  
   SSE 每到一个增量 chunk 不直接 setState，而是暂存到缓冲，在下一帧统一 append 到消息 `content`，避免频繁触发虚拟 DOM diff。
2. Map 结构加速定位（运行时索引）  
   `conversations` store 内部维护 `_convMap` 与 `_msgMap`（不持久化），O(1) 获取当前流式消息对象；持久化仍使用数组，保证序列化简单。
3. 增量 API：`appendChunkMessage` / `finishStreamMessage`  
   - `appendChunkMessage(id, chunk)`：把分片缓存在 `_streamBuffers`，下一帧刷新。  
   - `finishStreamMessage(id)`：确保最后一次 pending flush，并写入 `meta.streaming = false`。
4. 结束保障（flush）  
   即便流极快结束，也会在 `onFinish` 调用 `finishStreamMessage`，保证不丢最后一段文本。

### 10. 图片懒加载

项目提供了图片懒加载方案，确保聊天消息中的图片在进入视口时才加载，降低初始加载负担：

1. **IntersectionObserver API**  
   使用原生 `IntersectionObserver` 检测图片是否进入视口，支持 `rootMargin` 提前 100px 开始加载，提升用户体验。

2. **懒加载 composable**（`src/composables/useLazyImage.ts`）  
   - 提供 `useLazyImage()` 管理多个元素的观察
   - 提供 `useLazyImageState()` 管理单个图片的加载状态（loading/loaded/error）

3. **LazyImage 组件**（`src/components/LazyImage.vue`）  
   - 显示骨架屏占位符
   - 加载中显示 spinner
   - 加载失败显示错误状态
   - 加载完成渐入动画

4. **ChatMessage 集成**  
   消息组件中的图片使用 `data-src` 存储真实地址，进入视口时才赋值给 `src`。

### 11. 防抖与节流策略

使用 `lodash-es` 的 `debounce` 和 `throttle` 优化高频事件：

| 场景 | 方案 | 配置 | 效果 |
|------|------|------|------|
| 滚动事件（VirtualChatList） | throttle | 100ms, leading + trailing | 减少 90% 滚动计算 |
| 自动吸底检测（useAutoScroll） | throttle | 100ms, leading + trailing | 流畅响应用户滚动 |
| SSE token 追加 | requestAnimationFrame | 帧级合并 | 60fps 上限，避免卡顿 |

**为什么选择 lodash-es？**
- 经过大量生产环境验证，稳定可靠
- 支持 `leading`/`trailing`/`maxWait` 等丰富配置
- Tree-shaking 友好，只打包用到的函数
- TypeScript 类型定义完善

**工具文件**（`src/utils/optimize.ts`）：
```typescript
export { debounce, throttle } from 'lodash-es'
```
